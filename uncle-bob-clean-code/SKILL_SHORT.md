# Uncle Bob's Clean Code - 极简速查

## 1. SOLID 面向对象设计原则
*   **S (单一职责 - SRP)**：一个类应该只有一个引起它变化的原因。
*   **O (开闭原则 - OCP)**：对扩展开放，对修改关闭。依赖接口/抽象，而不是具体实现。
*   **L (里氏替换 - LSP)**：子类必须能够替换掉它们的父类，且不改变程序的正确性。
*   **I (接口隔离 - ISP)**：客户端不应被迫依赖它们不使用的方法，大接口应拆分为小接口。
*   **D (依赖倒置 - DIP)**：高层模块不应依赖低层模块，二者都应依赖抽象。

## 2. 干净的函数与命名规约
*   **函数核心法则**：
    *   **短小**：行数应极短（推荐 10 行以内）。
    *   **只做一件事**：每个函数只做一件事，且处于同一抽象层级。
    *   **参数控制**：最好是 0 或 1 个参数，严禁超过 3 个参数（多参数需打包成对象）。
    *   **禁止标志参数**：例如 `render(boolean isBold)` 是反模式，应拆为两个函数。
    *   **命令与查询分离 (CQS)**：一个函数要么改变状态，要么返回查询结果，不能二者兼施。
*   **注释治理**：注释是代码表达力失败的体现。“不要给糟糕的代码写注释，直接重构它”。
*   **防御空指针**：禁止返回 `null`，应返回空集合、`Optional` 或特殊空对象。

## 3. 整洁架构与测试规约
*   **依赖规则**：依赖只能单向向内指向核心业务策略（Entities/Use Cases）。外层（DB, Web 框架, UI）是内层的插件，严禁内层知道外层的任何细节。
*   **TDD 三定律**：
    1.  没有写失败的单元测试前，不能写任何生产代码。
    2.  测试代码只写到刚好失败为止，不能编译也算失败。
    3.  生产代码只写到刚好通过失败的测试为止。
*   **FIRST 原则**：单元测试必须是 **Fast**（快速）、**Independent**（独立）、**Repeatable**（可重复）、**Self-validating**（自我验证）、**Timely**（及时写入）。
*   **测试边界**：保证一个单元测试只测试一个概念，且保持生产代码与测试代码的同等清洁度。
