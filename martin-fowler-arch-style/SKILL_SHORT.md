# Martin Fowler 重构与架构 - 极简速查

## 1. 核心设计哲学
*   **可读性优先**：代码不仅是让计算机运行，首要是让人看懂。变量/方法名应当自我揭示意图。
*   **两顶帽子理论（Two Hats）**：写代码时必须分清状态：
    1.  **添加功能**：写新行为，编写测试（测试可能失败），不修改旧代码。
    2.  **重构**：调整代码结构（测试必须保持绿色），绝对不添加新功能。
    *   **原则：一次只戴一顶帽子，绝对不在同一个提交中混用。**
*   **演进式设计 (Evolutionary Design)**：不要试图做完美的超前设计（BDUF），坚持 YAGNI 原则，根据实际复杂度进行渐进式重构。

## 2. 核心重构手法与 Code Smells
*   **三原则 (Rule of Three)**：第一次做某件事直接做，第二次重复时感觉别扭，第三次出现重复时必须重构。
*   **关键 Code Smells 治理**：
    *   **Duplicated Code（重复代码）**：使用 `Extract Method` 提取出公用。
    *   **Long Method（过长方法）**：发现逻辑嵌套深或职责不单一，无条件重构，小方法能显著提高代码表现力。
    *   **Feature Envy（依恋情结）**：方法大量访问另一个类的数据，应当将该方法移动到那个类中。
    *   **Shotgun Surgery（散弹外科手术）**：修改一个逻辑需要修改多个文件，应通过封装把变化点聚合到一个类中。
*   **经典重构方法**：
    *   `Extract Method`（提取方法）、`Introduce Parameter Object`（引入参数对象）、`Replace Temp with Query`（用查询替换临时变量）、`Split Loop`（分拆循环）。

## 3. 技术债务消解
*   **债务四象限模型**：识别是“有意的/无意的”与“鲁莽的/审慎的”债务。
*   **重构时机**：
    *   预备性重构：加新功能前，先通过重构让新功能容易加入。
    *   认知性重构：看懂一段复杂的代码后，立刻重构让它更容易看懂。
    *   捡垃圾式重构：遇到杂乱的代码时顺手清理，留下比原来更干净的营地（童子军法则）。
